Προς διευκόλυνσή μας, αυτοί οι κατάλογοι μπορούν να θεωρηθούν flat, δηλαδή χωρίς υποκαταλόγους στο εσωτερικό τους.

Ωστόσο, και αυτό για τώρα μπορώ να το παραλείψω, αρχίζοντας να δουλεύω κατευθείαν στο πιο σημαντικό πρόγραμμα της εργασίας, τον `fss_manager`, στον οποίο θα κουμπώσω και την κονσόλα σε αργότερο στάδιο.

Για τώρα, επόμενος στόχος μου είναι να φτιάξω μια αρχική έκδοση του `fss_manager`, η οποία θα δέχεται ως παραμέτρους το `manager_logfile`, το `config_file` και το `worker_limits`, και θα κάνει κάποιες αρχικοποιήσεις.

Σημαντικό είναι να ξεκαθαρίσουμε από νωρίς τις δομές που θα χρησιμοποιεί ο `fss_manager` για την αποθήκευση και αναζήτηση πληροφοριών σχετικά με τους παρακολουθούμενους καταλόγους. Πιθανόν να φτιάξω μια απλή συνδεδεμένη λίστα, καθώς έχω λίγο καιρό να γράψω πρόγραμμα σε C και η λίστα είναι πιο απλή δομή από το hash table. Αν και λιγότερο αποδοτική, δεν περιμένουμε το σύστημα να διαχειρίζεται τεράστιο αριθμό καταλόγων ούτως ή άλλως.

### Dev Log: Beginning the Real Work (a.k.a. Μπαίνουμε στα βαθιά)

Έχω καλύψει, έστω και μερικώς, τους αρχικούς στόχους που είχα θέσει, και πλέον έχω μια μέτρια αλλά λειτουργική βάση. Από εδώ και πέρα όμως ξεκινά το πιο “βαρύ” κομμάτι της εργασίας, αφού πρέπει να μπω στην υλοποίηση των αυστηρότερων απαιτήσεων της εκφώνησης, και ταυτόχρονα να ξεκαθαρίσω αρκετά τεχνικά σημεία.

Αυτή τη στιγμή, ο `fss_manager` κάνει την αρχικοποίηση των watchers για τους καταλόγους που προκύπτουν από το `config_file`, κάτι που φαίνεται λογικό. Αν και στην αρχή είχα την εντύπωση πως ίσως αυτή τη δουλειά έπρεπε να την κάνουν οι workers, διαβάζοντας ξανά την εκφώνηση, καταλαβαίνω πως οι workers απλά εκτελούν tasks συγχρονισμού — δεν έχουν ευθύνη για inotify watches ή updates στη λίστα των παρακολουθούμενων directories. Αυτά είναι καθαρά ευθύνη του `fss_manager`.

Οπότε, αποφάσισα το εξής: ο `fss_manager` θα ξεκινά άμεσα την παρακολούθηση με `inotify`, και κάθε φορά που θα ανιχνεύεται αλλαγή ή θα διαβάζεται νέο ζεύγος από το config, θα δημιουργείται ένα task για συγχρονισμό. Αυτά τα tasks θα μπαίνουν σε μια ουρά, και οι workers θα τα τραβάνε όταν μπορούν. Αν για παράδειγμα ξεκινήσει ένας Full Sync worker και πριν τελειώσει γίνει νέα αλλαγή στο source, θα μπει νέο task, χωρίς να χαθεί η αλλαγή.

Αυτό που θέλω να προσέξω — και άρχισα ήδη να το σκέφτομαι πιο σοβαρά — είναι τα **race conditions**. Οι workers μπορούν να δουλεύουν παράλληλα σε διαφορετικά αρχεία, ακόμα και στον ίδιο κατάλογο, αλλά **δεν πρέπει να δουλεύουν στο ίδιο αρχείο ταυτόχρονα**. Έψαξα λίγο και βρήκα τη `flock()`, η οποία φαίνεται να είναι αυτό που χρειάζομαι: ένας απλός μηχανισμός για αποκλειστική πρόσβαση σε ένα αρχείο. Έτσι, κάθε worker που θέλει να συγχρονίσει ένα αρχείο θα προσπαθεί να πάρει αποκλειστικό lock με `flock()` πριν κάνει οτιδήποτε. Αν αποτύχει, απλά θα retryάρει αργότερα.

Πέρα από αυτό, δεν νομίζω ότι χρειάζεται ιδιαίτερος συγχρονισμός ανάμεσα στους workers. Ένα θέμα που ίσως θέλει λίγο προσοχή είναι η συγγραφή στα logs — δε θα ήθελα να ανοίγουν όλοι ταυτόχρονα το log file. Θα φροντίσω κι αυτό να γίνεται με κάποιο μηχανισμό αποκλεισμού ή απλά με central logging μέσω του `fss_manager`.

Επόμενο βήμα: να φτιάξω τον κώδικα για τους workers. Θα ξεκινήσω από τη λογική του Full Sync και σιγά-σιγά θα προσθέτω τις υπόλοιπες λειτουργίες (`ADDED`, `MODIFIED`, `DELETED`). Παράλληλα θα ενσωματώνω και τον συγχρονισμό με `flock()` ώστε να είναι προστατευμένη η πρόσβαση σε κάθε αρχείο.

### Dev Log: Ολοκλήρωση Backend και Προετοιμασία για το Frontend

Έχω ολοκληρώσει ένα πολύ μεγάλο μέρος του backend, με σχεδόν όλες τις βασικές λειτουργίες να δουλεύουν σωστά. Αυτό που απομένει είναι να φτιάξω τον command handler: αρχικά θα στέλνω εντολές απευθείας στο named pipe με `echo "command" > fss_in` για να ελέξω τη σωστή λειτουργία του handler. Στη συνέχεια, θα προχωρήσω στην ανάπτυξη ενός command line frontend, όπως περιγράφεται στην εκφώνηση, μέσω του οποίου θα μπορούν να στέλνονται εντολές, να λαμβάνονται αποτελέσματα, και να γίνεται σωστή καταγραφή μέσω μηχανισμού logging.

Παράλληλα, εντόπισα ένα σημαντικό ζήτημα σχετικά με τους workers. Αυτή τη στιγμή διαβάζω δεδομένα από τους workers μόνο όταν ολοκληρώνουν τη δουλειά τους. Όμως, σε ακραίες περιπτώσεις, αν ένας worker χρειαστεί να γράψει πάρα πολλά δεδομένα και ξεπεράσει το όριο του pipe buffer του συστήματος, τότε θα μπλοκάρει μέχρι να διαβαστούν κάποια δεδομένα. Επειδή το δικό μου μοντέλο διαβάζει μόνο στο τέλος, αυτό μπορεί να οδηγήσει σε αδιέξοδο (deadlock). Προς το παρόν, επειδή τα πειράματα που κάνω έχουν λογικό μέγεθος, δεν αναμένεται να προκύψει πρόβλημα. Ωστόσο, το σημειώνω για να το διορθώσω αργότερα: θα χρειαστεί πιο σύνθετο parsing ώστε να υποστηρίζεται ανάγνωση τμηματικών δεδομένων και αποθήκευση της προόδου μέχρι να ολοκληρωθεί η πλήρης ανάγνωση.

Στόχος για τις επόμενες ώρες είναι να ολοκληρώσω το frontend καθώς και το απαιτούμενο shell script, ώστε να έχω μια πλήρη και λειτουργική εργασία. Μετά από αυτό, αν υπάρχει χρόνος, θα ασχοληθώ με τη βελτίωση για το προαναφερθέν edge case, ώστε να τελειοποιηθεί το σύστημα.
